# Remediate the Codex Security Findings

## Summary

Implement the remediation as four reviewable Git Town slices from current `origin/main`: permission boundaries, trusted URLs/media, Stripe ingress/binding, and registration concurrency. Add one patch-level Knope changeset and write final evidence to the scan’s `artifacts/fix_report.md`.

## Implementation Changes

### Authorization and permissions — DSC-001, DSC-006, DSC-016

- Reuse one event-organizer predicate for `events.canOrganize` and `events.getOrganizeOverview`. Require a tenant user and organizer, `events:organizeAll`, or `finance:manageReceipts` authority before querying attendee data; return typed unauthorized/forbidden errors.
- Introduce `TenantRolePermission` and its schema, excluding `globalAdmin:*` and `globalAdmin:manageTenants`. Use it for role RPC inputs, outputs, and persisted role typing while retaining the broader effective-permission schema for genuine platform principals.
- At request-context resolution, discard only platform-global literals found in tenant roles, preserve legitimate tenant permissions, and emit structured security warnings. Supply audit/cleanup SQL, but never mutate deployed data automatically.
- Keep `users:assignRoles` unrestricted, including self-assignment, because it is now explicitly full tenant-administrator authority. Update its label, description, role-form warning, product guidance, and generated documentation so assigning this capability cannot be mistaken for limited delegation.

### Trusted URLs, receipts, and icons — DSC-002, DSC-008, DSC-019

- Add required `tenant.canonicalRootUrl`, administered only through global-admin tenant create/edit. Normalize it to an HTTPS origin whose hostname exactly matches the primary domain; reject credentials, paths, query strings, fragments, non-default ports, and mismatches.
- Build Stripe and email links exclusively from the tenant canonical root. Permit `BASE_URL` only as an explicit loopback development override; remove all authority from `Origin`, `Host`, and forwarded headers and fail before side effects when configuration is invalid.
- Add a persisted `finance_receipt_uploads` preflight linked uniquely and non-nullably from each receipt. Upload returns only `{uploadId}`; submission accepts `{fileName, uploadId}` and atomically consumes a matching uploaded tenant/event/user record. Preview signing must use the bound upload row and return no URL when association or key scope is invalid.
- Change `icons.add` to accept `{icon, usage}` where usage identifies event create/edit, template create/edit, or category management. Enforce the same server-side ownership/capability rule that grants access to that dialog; allow explicit audited platform-admin access.
- Accept only normalized valid Icons8 identifiers: ASCII alphanumeric/hyphen segments, one optional colon, 128 characters total and 96 per segment. Verify the fixed-host response is a bounded 128×128 PNG before insertion.
- Bound expensive icon work with a five-second fetch timeout, 256 KiB streamed response cap, at most two concurrent resolutions, and a 256-entry LRU cache with success/failure TTLs. Limit search input to 64 characters, escape SQL wildcards, and return at most 50 results.
- Keep the catalog lifetime unlimited and add no actor/tenant rate throttle, quota, or eviction, per the chosen product policy. Record persistent catalog growth and sustained valid-icon traffic by authorized authors as accepted residual risk.

### Stripe and registration integrity — DSC-004, DSC-010, DSC-018, DSC-020

- Read Stripe webhook bodies incrementally with an early `Content-Length` check and a 200 KiB route-local cap. Cancel the stream immediately on overflow and return 413 without signature work; do not lower the global server/RPC body limit.
- Resolve completed and expired Checkout sessions from persisted `stripeCheckoutSessionId`, not metadata. Require exact tenant, registration, transaction, connected-account, method/type/status, and PaymentIntent agreement. Metadata is corroboration only.
- Require exactly one pending transaction update before confirming or cancelling a registration. Return 400 for invalid bindings and 409 for state races, leaving registration, payment, capacity, and add-ons unchanged.
- Add a partial unique registration index on `(tenantId, eventId, userId)` where status is not `CANCELLED`. Lock the tenant membership before registration/waitlist/transfer decisions, recheck inside the transaction, and map losing inserts to the existing domain conflict.
- Add a partial unique pending-payment index on `(tenantId, eventRegistrationId)` for pending registration transactions.
- For paid manual approval, create the pending transaction claim and reserve capacity/add-ons atomically before calling Stripe. Use a claim-derived idempotency key, bind the returned session conditionally, and enqueue the email with that binding.
- If session persistence fails, expire the Stripe session before releasing the claim and reservations. Ambiguous Stripe outcomes retain the claim and fail closed, preventing a second payable session.

## Assumptions and Rollout

- Follow the relaunch policy: update the new schema directly and add no repository migration steps. Existing-data transfer, if required, remains a separate TypeScript job.
- Before applying indexes to a persistent environment, audit for duplicate active registrations and pending transactions. Stop for manual Stripe-aware reconciliation if any exist; never auto-select a winner.
- Existing poisoned tenant roles are neutralized at runtime before cleanup. Production database inspection or cleanup requires separate explicit authorization.
- Failed or abandoned receipt uploads remain an operational storage-cleanup follow-up; they cannot be consumed or presigned.
- `users:assignRoles` is intentionally full tenant-admin authority.
- Icon catalogs are unlimited and unthrottled over time; only authorization, validity, per-request bounds, concurrency, search bounds, and cache bounds are enforced.
